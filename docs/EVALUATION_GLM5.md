## 0. Core Insight

**TerAgent's security architecture is fundamentally an "anti-AI-self-destruction" system.**

The threat source is not a malicious attacker, but the uncontrollable behavior produced by AI models under hallucination — AI "doing harm with good intentions": saying "read config" but executing `chmod`, saying "optimize code" but deleting critical imports, saying "install dependencies" but running `pip install` on untrusted packages.

This insight fundamentally changes the evaluation framework: traditional security concerns (TOCTOU, injection bypasses) have lower priority; intent-action consistency, conservative defaults, and circuit-breaker stop-loss are the core defenses.

## 3. Architecture Design — 9.0/10

### Highlights

**TAP IR is the project's most outstanding abstraction.** Borrowing LLVM's IR philosophy, it transforms the complex requirement of "same intent, different optimal prompts per model" into a clean compilation pipeline:

```
TAPRequest → Compiler.compile() → CompiledPrompt → Adapter.send() → TAPResponse
```

**4×3 orthogonal composition is a textbook OCP practice:**

| | OpenAI Adapter | Anthropic Adapter | Mock Adapter |
|---|---|---|---|
| DefaultCompiler | ✅ | ⚠️ Mode degradation | ✅ |
| GLMCompiler | ✅ | ⚠️ | ✅ |
| AnthropicCompiler | ⚠️ | ✅ | ✅ |
| DeepSeekCompiler | ✅ | ⚠️ | ✅ |

New Compiler: 1 file + 1 registration line. New Adapter: same. New combination: **0 lines of code**.

**Design patterns used appropriately:** Strategy (Compiler/Adapter are replaceable), Template Method (`_inject_context` shared logic), Registry (self-registration mechanism), Factory (`create_provider`), Facade (`__init__.py` lazy imports), Observer (EventBus), Decorator (fallback provider), Builder (Message factory method group) — 8 patterns, none for showboating, each solving a real problem.

**Strict unidirectional layered decoupling:** Zero dependency between Compiler and Adapter — they interact only through the `CompiledPrompt` data structure. `ModelProvider` composes both via constructor injection, doing neither compilation nor sending, only orchestration.

### Shortcomings

| Issue | Location | Impact |
|-------|----------|--------|
| `create_provider()` has hardcoded Adapter conditional branches | `__init__.py:558-576` | New Adapter requires modifying this function, violating OCP |
| Mode A/B mutual exclusivity has no compile-time guarantee | `tap.py:106-146` | When both fields are populated, `mode` silently returns "messages", potentially losing data |
| Mode A↔B conversion logic duplicated 6 times | 3 methods × 2 Adapters | High maintenance burden |
| FAKE_TOOLS completely duplicated in two Adapters | `openai_compatible.py` / `anthropic_native.py` | Violates DRY |
| `TAPRequest.meta/context` are plain dict | `tap.py:34-35` | No runtime validation; typos pass silently |

## 4. Anti-Hallucination Security — 7.5/10

### Core Judgment: Direction Correct, Missing Key Layer

TerAgent's security is designed around "AI may produce dangerous behavior" — direction entirely correct:

| Defense Line | Hallucination Scenario It Guards Against | Rating |
|-------------|------------------------------------------|:------:|
| Default DENY | AI doesn't know what it can't do | ✅ |
| 7-layer permission resolution | AI forgets previous constraints | ✅ |
| 6-layer command defense | AI "does harm with good intentions" (says "clean up" but `rm -rf`) | ✅ |
| 2PC file writes | AI overwrites important files / writes wrong format | ✅ |
| 3-level sandbox degradation | AI executes uncontrollable code in dangerous environments | 🟡 |
| 4 circuit breakers | AI infinite loop / infinite retry / no progress | ✅ |
| AI permission classifier | Cross-validates decisions with another AI | 🟡 |
| **Intent-action consistency check** | AI says one thing but does another | ❌ Missing |

### Detailed Evaluation by Subsystem

#### ✅ Permission System — First Wall Against Hallucination

**Correct design:** AI under hallucination doesn't know its boundaries; default DENY ensures AI can only do what's explicitly allowed. Rule priority (user > config > project > system > default) handles AI "forgetting constraints." Same-priority DENY takes precedence over ALLOW — conservative strategy against hallucination.

**AI Permission Classifier re-examined:** The risk isn't "attacker injection" but **classifier self-hallucination** — but structured input + restricted output + conservative default significantly reduces this risk. Low-confidence forced DENY is the correct strategy to counter hallucination.

**Shortcomings:**
- PLAN level allows all write tools (`permission.py:542-561`); hallucinating AI may think "I'm in planning phase, everything is allowed"
- `check_tool_params` path extraction key names incomplete (`permission.py:577`); AI may use `target`/`destination` etc. and bypass checks — not malicious bypass, but AI wording inconsistency

#### ✅ Command Defense — Directly Intercepts AI "Good Intentions, Bad Results"

6-layer defense precisely covers typical AI hallucination behaviors:
- AI says "clean project" → `rm -rf /` → Layer 3 blacklist intercepts
- AI says "install dependencies" → `curl | sh` → Layer 5 cross-chain detection intercepts
- AI says "run with root" → `sudo ...` → Layer 3 privilege escalation intercepts
- AI says "run this script" → `python3 -c "..."` → Layer 3 inline script intercepts

**Shortcomings:**
- `python3 /tmp/malicious.py` doesn't trigger blacklist — AI may generate a script file then execute it, bypassing all 6 layers
- `classify_command_risk` and `check_command_safety` are two inconsistent logic sets — DangerousCommandHook and sandbox use different rules
- **Missing Layer 7: Intent-action consistency check** — AI says "I'll read the config" but executes `chmod 777 /etc/config`; current system cannot detect this inconsistency

#### ✅ 2PC File Writes — Prevents AI Overwrite / Format Errors

**Correct design:** AI hallucination file operation risks — overwriting entire file leaving one line, writing YAML as JSON, deleting critical imports — 2PC + read-before-write contract + backup rollback addresses exactly these.

Under the anti-hallucination model, "partial commit window" risk is significantly reduced: AI won't maliciously read during an "attack window"; it just writes wrong content under hallucination, and backup rollback can fix it.

#### 🟡 Sandbox Degradation — Needs Architecture Constraint, Not AI Choice

Under anti-hallucination model, the core risk of degradation isn't "attacker causes environment downgrade" but **AI proactively choosing a lower security level when encountering difficulties**:
- AI fails in Firecracker → hallucination makes it think "sandbox has problems" → bypass sandbox and execute directly
- AI generates a timeout script → decides "skip sandbox, just run it"

**Shortcomings:**
- Degradation only logs `logger.warning`; no audit log, no user notification, no forced confirmation
- AI should not have the authority to decide on degradation — this should be an architectural constraint, not a runtime choice

#### ✅ Circuit Breakers — Specifically Designed for AI Hallucination

| Breaker | Hallucination Scenario It Guards Against | Rating |
|---------|------------------------------------------|:------:|
| ConsecutiveFailureBreaker | AI repeatedly retries an operation destined to fail | ✅ Standard closed→open→half_open implementation |
| CostBudgetTracker | AI infinitely consumes tokens under hallucination | 🟡 Hard limit not enabled by default |
| LatencyBreaker | AI stuck on a slow operation, keeps retrying | 🟡 Warning only, no hard block |
| **ProgressDetector** | AI loops "appears to be doing something" with no progress | ✅ **Specifically designed for AI hallucination** — humans don't do 10 steps with no progress, but AI does |

**Shortcomings:**
- `_consecutive_failures` is actually "total failures within window" not "consecutive" failures; name doesn't match semantics
- `CostBudgetTracker` default `enable_hard_limit=False`; hallucinating AI can infinitely consume tokens

#### 🟡 Audit System — Post-Hoc Diagnostic Evidence for Hallucination

**Audit logs are more important in anti-hallucination scenarios than in traditional security** — they are the sole basis for post-hoc diagnosis of "why did the AI hallucinate."

**Shortcomings:**
- Silent discard on write failure (`audit.py:136-144`), no local file fallback — if AI hallucination causes database corruption, diagnostic evidence is lost
- Audit records are unsigned/unencrypted; AI could theoretically modify records (though AI won't "deliberately cover up," it may execute an operation that overwrites the audit database while hallucinating)

## 5. Streaming Execution & Context Management — 7.5/10

### Streaming Execution

**Dual-track dispatch design is elegant:** immediate (read_only + concurrency_safe → immediate async execution) + deferred serial (write operations → serial execution after stream ends), significant latency optimization.

**ToolCallAccumulator's two-level JSON completeness detection** is a performance optimization highlight: first uses brace balance for fast screening, then `json.loads` for precise validation, avoiding expensive parsing on every delta.

**Shortcomings:**
- Anthropic Adapter's `stream()` is incompatible with `StreamingToolExecutor`; Anthropic native protocol cannot use streaming tool calls
- Missing partial result handling after stream interruption — queued but unexecuted tool calls only logged
- Nested JSON parameters may cause brace balance misdetection, prematurely triggering `TOOL_CALL_COMPLETE`

### Context Management

**AutoCompactor's triple protection is solid:** circuit breaker (stop after 2 consecutive failures) + count limit (max 5 per session) + message count floor (too few → don't compress).

**Microcompactor's per-tool-type differentiated compression** is a highlight: read_file keeps head 20 + tail 10 lines, design/plan extracts title + first line content, safe fallback (if compression makes output longer, degenerate).

**Shortcomings:**
- CJK-mixed heuristic token estimation has insufficient precision (`context_window.py:106-110`), `_last_estimated_tokens` cache has no invalidation mechanism
- AutoCompactor's `RETAIN_COUNT=8` hardcoded without considering individual message size; 8 large messages may still exceed budget
- LLM summarization cost/benefit ratio unevaluated — LLM call tokens consumed may exceed tokens saved

## 6. Sub-Agent Coordination & Event System — 7.0/10

### SubAgentManager

**Step budget + tool whitelist + three execution modes** covers basic scenarios.

**Serious issues:**
- Concurrency limit check TOCTOU (`sub_agent_manager.py:274-289`): modifies state before checking, effective limit is MAX+1
- FORK mode is an empty shell: delegates to `_run_sync`, no KV cache optimization, only a semantic marker
- `_active_agents` retains last 100 completed agents; long-running memory growth

### EventBus

**Pub-sub design is solid:** sync/async dual-channel, error isolation, `once()` semantics, `wait_for()` conditional wait.

**Shortcomings:**
- `once()` `_fired` flag may fail under concurrent calls
- `_shared` dict marked DEPRECATED but not removed
- Event history truncation is non-atomic

### Intent Classification

**Three-layer funnel design is elegant:** deterministic rules (zero cost) → heuristic (zero cost) → LLM (low cost). Keyword conflict uses "fix before build" safety-first strategy.

**Shortcoming:** LLM classification failure defaults to `CREATE_PROJECT` (heaviest pipeline); should default to `CHAT`.

### Session Persistence

**Atomic write + thread safety + memory cache** design is correct.

**Shortcomings:**
- `_cache` has no size limit; long-running memory continues to grow
- `auto_save` triggers disk write per message; rapid conversation causes frequent I/O

## 11. Final Assessment

### One Sentence

> **TerAgent gives the best-in-class answer to "how should AI do things," but is missing one layer for "what happens when AI does things wrong."**

### Detailed Judgment

**At the architecture level:** TAP IR + Compiler×Adapter orthogonal composition is the most elegant abstraction in the AI Agent middleware space. Layered decoupling, design pattern usage, and extensibility are all textbook-grade. 9.0 is well-deserved.

**At the security level:** The default-deny/conservative degradation/circuit-breaker stop-loss direction is completely correct, precisely targeting the "AI hallucination" real threat. But it's missing the most critical layer — **intent-action consistency check**. The core characteristic of hallucination is "says one thing, does another," and the system only checks "what was done." This is the key leap from 7.5 to 9.0.

**At the engineering level:** Test coverage is decent but AgentLoop blind spot + no CI/CD are hard flaws; documentation depth is excellent but has runtime-error code examples. 6.5-7.0 is reasonable.

**Overall:** As a 0.0.1 Alpha, TerAgent demonstrates **excellent architectural design taste** and **correct security intuition** (though the author may not have explicitly realized the threat model is "anti-hallucination" rather than "anti-attacker"). After completing Phase 1-2, the project can reach production-ready status. The core innovation (TAP IR orthogonal compilation) has the potential to become an industry standard.
